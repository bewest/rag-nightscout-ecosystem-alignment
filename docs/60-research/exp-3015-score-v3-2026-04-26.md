## Goal

Wire all three Phase 3 findings into the autoresearch fitness function used by `aid-autoresearch` to rank candidate controller configurations:

1. **Per-patient (T*, M*) recommendations** from EXP-3012 (`exp-3012_per_patient.parquet`).
2. **Carb-aware trough proxy** from EXP-3014 as the new default (`--proxy carb_aware`).
3. **Braking-ratio phenotype gate** from EXP-3013 — drop patients with `braking_ratio >= 0.10` (default), since they are already saturated and cf-replay finds no headroom for them.

## CLI shape

```
cf_replay_score_v3.py
    --smb-multiplier 0.5            # cohort-uniform M (default 1.0)
    --t-shift 30                    # cohort-uniform T (default 0.0)
    --per-patient                   # use exp-3012 per-patient (T*, M*); ignores cohort knobs
    --proxy {carb_aware,worst_case} # default carb_aware (EXP-3014)
    --braking-gate [VALUE]          # default 0.10 if flag given, no gate if omitted
    --json                          # machine-readable output
```

Backwards-compatible: invoked without new flags it reduces to the v2 cohort-multiplier behaviour but with the carb-aware proxy as the default trough model.

### Observations

1. **Baseline (current observed) FAILS** the 1 % hypo gate under the carb-aware proxy. This is correct: the v3 scorer reading observed dosing identifies the ascent-window tail risk that motivates the program. (Note: the proxy's *observed* hypo rate is not the patient's *clinically observed* hypo; it's the cf-replay forecast under current dosing.)

2. **Frontier (uniform mult=0.5, T=30) passes** with the largest score, +0.020 over baseline (≈ +2.9 % composite improvement). This is the **deployment-ready cohort recommendation**.

3. **Per-patient mode passes** but scores marginally *lower* than uniform-frontier on the cohort composite. This is by design: per-patient mode applies (T*, M*) only to the 9 509 events on responsive patients (53 % of cohort). The cohort composite weights all controllers equally, so dropping high-braking AAPS-oref0 events (3336 → 123) hurts the cohort mean even though it improves per-responsive-patient outcomes. The clinically-correct interpretation: per-patient mode is for individualised tuning; uniform-frontier is for population deployment.

4. **Braking gate behaves as designed**: drops 8 410 events (47 %), most of which are AAPS-oref0 (high-braking, no SMB) and Loop-AB-OFF. The remaining cohort is the actionable subset.

## Backward compatibility

`cf_replay_score_v2.py` is unchanged on disk; existing autoresearch wiring continues to work. v3 is a parallel script. If/when the autoresearch driver is updated to consume v3, the recommended invocation is:

```bash
# population deployment (replicates Phase 2 closeout)
cf_replay_score_v3.py --smb-multiplier 0.5 --t-shift 30 --json

# individualised analysis (Phase 3 closeout)
cf_replay_score_v3.py --per-patient --braking-gate --json
```

## Implementation notes

- The kernel-direction bug discovered while writing EXP-3014 (`t_peak = half - T` vs the correct `half + T`) was carried over from EXP-3011 and is fixed in v3 to match the corrected EXP-3014 formula.
- `_isf_map()` is identical to v2's logic.
- The smoke-test compiles a regression bound that catches: (a) future proxy regressions that flip baseline-vs-frontier ordering; (b) braking-gate not loading the phenotype parquet; (c) carb-aware proxy not being applied.
- Per-patient mode falls back to cohort knobs (`--smb-multiplier`, `--t-shift`) for any patient missing from `exp-3012_per_patient.parquet`. This makes the mode compose-ably safe for new cohorts.
